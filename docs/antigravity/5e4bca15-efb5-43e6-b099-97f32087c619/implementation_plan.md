# Firestore Alert Storage — Implementation Plan

Store every `/api/webhook/alert` payload in Cloud Firestore after delivery, capturing full request context, enrichment, delivery results, and token usage.

## Proposed Changes

---

### New: Firebase Admin SDK client

#### [NEW] `src/lib/firebaseAdmin.js`

Lazy-initializes `firebase-admin` once using env-based credentials:
- **`GOOGLE_APPLICATION_CREDENTIALS`** — path to service-account JSON file (file-based auth, used locally or on VMs)
- **`FIREBASE_SERVICE_ACCOUNT_JSON`** — inline JSON string (preferred for Render secret env vars)
- **`FIREBASE_PROJECT_ID`** — explicit project override (optional when the JSON already contains it)

Exports a `getFirestore()` helper that returns the Firestore `db` instance (singleton, safe to call many times).
Initialization errors are logged as warnings and return `null` — the alert flow **never fails** if Firestore is unavailable.

---

### New: AlertStorageService

#### [NEW] `src/services/storage/AlertStorageService.js`

A service class that wraps Firestore document creation. Implements fail-open: any Firestore write error is caught, logged as `console.warn`, and the calling code is unaffected.

**Collection**: `alerts`
**Document ID**: auto-generated by Firestore (`addDoc` equivalent in Admin SDK → `collection.add()`)

**Document schema** (all stored in a single `alerts/{docId}` document):

| Field | Type | Description |
|---|---|---|
| `receivedAt` | Timestamp | Server-side time when the alert was received (set via `FieldValue.serverTimestamp()`) |
| `text` | string | Original alert text (from `alert.text`) |
| `enriched` | boolean | Whether Gemini/TradingView enrichment ran |
| `enrichmentData` | map \| null | `alert.enriched` object (sentiment, insights, technical_levels, sources, etc.) |
| `tokenUsage` | map \| null | `tokenUsage.toJSON()` result including model breakdown and `formattedSummary` |
| `deliveryResults` | array | Array of per-channel `SendResult` objects from `notificationManager.sendToAll()` |
| `source` | string | Fixed value `"webhook"` |
| `useTradingViewData` | boolean | Query param flag from the request |

### Modified: Alert webhook handler

#### [MODIFY] `src/controllers/webhooks/handlers/alert/alert.js`

After the `notificationManager.sendToAll()` call resolves (in the happy path inside `postAlert`), fire-and-forget save to Firestore:

```js
// Fire-and-forget — never blocks the HTTP response
alertStorageService.saveAlert({
  text: alert.text,
  enriched,
  enrichmentData: alert.enriched || null,
  tokenUsage: tokenUsageJSON,
  deliveryResults: results,
  useTradingViewData,
}).catch(() => {}); // errors already logged inside the service
```

The HTTP response (`res.json(...)`) is **not delayed** by the Firestore write.

### New: Firestore security rules

#### [NEW] `firestore.rules`

Server-side (Admin SDK) writes bypass security rules, so rules only need to protect **client-side** reads. Since this is a server-only bot, the rules will deny all client reads/writes:

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    // All access is server-side only (firebase-admin SDK)
    // No client SDK is used, so deny all client access.
    match /{document=**} {
      allow read, write: if false;
    }
  }
}
```

## Open Questions

> [!IMPORTANT]
> **Firebase Project ID**: The `firebase-admin` SDK needs to know which project to write to. This is typically embedded in the service account JSON. If you use `FIREBASE_SERVICE_ACCOUNT_JSON`, please make sure the JSON you paste includes `"project_id"`. Alternatively, set `FIREBASE_PROJECT_ID` as a fallback.

> [!NOTE]
> **Firestore database ID**: By default the Admin SDK uses the `(default)` database. If your Firestore instance uses a custom database ID (e.g. `cabros-bot`), let me know and I'll add a `FIREBASE_DATABASE_ID` env var.

> [!NOTE]
> **`firebase-admin` version**: The project already has `firebase-admin@^9.8.0`. The Admin SDK 9.x API is stable and compatible with what's described above. No version upgrade needed.
